## Назначение

Документ описывает проблему, при которой локальные или внешние облачные субагенты видят задачу, но не могут создавать или редактировать файлы из-за настроек оболочки.

Главная мысль: модель может уметь писать файлы, но право на запись выдаёт не модель, а среда запуска: Codex, OpenCode, sandbox, permissions и доступные tools.

Базовый режим для субагентов: работать с локальными файлами на этом ПК через Codex/OpenCode-инструменты, без Git. Git не используется для чтения, сравнения, поиска или изменения файлов, если пользователь отдельно не дал Git-задачу и не разрешил конкретные read-only Git-команды.

## Справочные копии настроек

Это центральный документ. Для понятного Obsidian-графа связи сделаны звездой: от этого документа идут отдельные ветки к справочным копиям настроек.

Справочные копии:

- [[Настройки субагентов/Codex_simple_local|Codex simple_local]] — копия и пояснение для `C:\Users\REX\.codex\agents\simple_local.toml`;
- [[Настройки субагентов/OpenCode_global_config|OpenCode global config]] — копия и пояснение для `C:\Users\REX\.config\opencode\opencode.jsonc`;
- [[Настройки субагентов/OpenCode_qwen30b_local|OpenCode qwen30b local]] — копия и пояснение для `C:\Users\REX\.config\opencode\agents\qwen30b-local.md`;
- [[Настройки субагентов/OpenCode_qwen480b_cloud|OpenCode qwen480b cloud]] — копия и пояснение для `C:\Users\REX\.config\opencode\agents\qwen480b-local.md`.

Важно: справочные копии не являются рабочими конфигами. Реальные настройки искать и менять нужно по оригинальным путям, указанным сверху в каждой справочной заметке.

## Рабочие агенты

Локальный агент Codex:

```text
simple_local / qwen3-coder:30b / локальный
```

Локальный агент OpenCode:

```text
qwen3-coder:30b / OpenCode Desktop / локальный через чат
```

Управление локальным агентом OpenCode:

```text
qwen3-coder:30b / профиль qwen30b-local
```

с этим агентом работает основной Codex. Пользователь может попросить Codex поставить ему задачу и может перенести подготовленную команду в OpenCode-чат, но постановку задачи, scope, проверку результата и итоговый отчёт делает основной Codex.

Внешний облачный агент OpenCode:

```text
qwen3-coder:480b-cloud / OpenCode Desktop / внешний через чат
```

Агенты OpenCode могут иметь локальный доступ к файлам через OpenCode Desktop, если OpenCode запущен в нужной рабочей папке и permissions разрешают чтение/запись.

## Текущий статус проверки

Проверено после настройки и перезапуска:

- `qwen3-coder:480b-cloud / OpenCode Desktop / внешний через чат` — успешно создал тестовый файл и записал отчёт в `Задача_внешнему_субагенту.md`;
- `qwen3-coder:30b / OpenCode Desktop / локальный через чат` — профиль создан, ожидает теста записи после выбора этого агента в OpenCode;
- `qwen3-coder:30b / Codex subagent / local` — controlled keep-warm успешен: агент вернул `READY`;
- `qwen3-coder:30b / Codex subagent / local` — тест записи неуспешен: файл не создан, агент не смог успешно использовать файловый инструмент записи в Codex-subagent-сессии;
- `simple_local / qwen3-coder:30b / локальный` — чтение, анализ и controlled keep-warm использовать можно, но создание файла через Codex-subagent пока считать неподтверждённым.

Вывод: облачный агент через OpenCode сейчас подтверждён как локально пишущий под контролем permissions. Локальный qwen30b через Codex-subagent подтверждён для keep-warm, но не подтверждён для записи. Для проверки записи локальной модели `qwen3-coder:30b` использовать отдельный маршрут OpenCode Desktop с профилем `qwen30b-local`.

## Симптом проблемы

Субагент получает задачу создать файл, но отвечает примерно так:

```text
write_file недоступен
mcp__node_repl недоступен
не могу создать или изменить файл в этой среде
```

Это не обязательно значит, что модель не умеет писать. Чаще это значит:

- роль агента запущена в `read-only`;
- OpenCode/Codex не выдал инструмент записи;
- конфиг изменён, но приложение ещё не перезапущено;
- агент пытается использовать неправильный инструмент;
- OpenCode ждёт подтверждение пользователя на запись.

## Настройка Codex local subagent

Проверить файл:

```text
C:\Users\REX\.codex\agents\simple_local.toml
```

Для локального `qwen3-coder:30b` должен быть режим:

```toml
sandbox_mode = "workspace-write"
```

В инструкциях агента должно быть явно сказано:

- писать только по прямому разрешению основного агента;
- менять только явно указанные файлы и каталоги;
- не выходить за scope;
- работать локально на ПК с файлами, а не через Git;
- не использовать Git вообще для обычных подзадач и тестов записи;
- перечислять все изменённые файлы в отчёте.

Если было `sandbox_mode = "read-only"`, локальный субагент может читать, но не сможет нормально создавать файлы.

## Настройка OpenCode cloud agent

Проверить глобальный конфиг OpenCode:

```text
C:\Users\REX\.config\opencode\opencode.jsonc
```

Минимально безопасные permissions:

```json
"permission": {
  "edit": "ask",
  "bash": "ask",
  "external_directory": "ask"
}
```

Так OpenCode может работать с локальными файлами, но спрашивает пользователя перед записью и shell-командами.

Для OpenCode-субагентов действует то же правило: они работают с локальными файлами на ПК. Git не используется вообще, если текущая задача отдельно не разрешила конкретные read-only Git-команды.

Проверить профиль внешнего облачного агента:

```text
C:\Users\REX\.config\opencode\agents\qwen480b-local.md
```

Профиль должен указывать:

```text
model: ollama/qwen3-coder:480b-cloud
permission:
  edit: ask
  bash: ask
  external_directory: ask
```

Если OpenCode спрашивает разрешение, пользователь подтверждает только те действия, которые прямо входят в задачу.

## Настройка OpenCode local agent

Проверить профиль локального агента OpenCode:

```text
C:\Users\REX\.config\opencode\agents\qwen30b-local.md
```

Профиль должен указывать:

```text
model: ollama/qwen3-coder:30b
permission:
  edit: ask
  bash: ask
  external_directory: ask
```

Этот маршрут нужен для проверки той же локальной модели `qwen3-coder:30b`, но не через Codex-subagent, а через OpenCode Desktop. Если Codex-субагент не получил файловые tools, OpenCode всё равно может дать модели локальную запись через свои инструменты и подтверждения пользователя.

Тест этого маршрута не требует Git. Агент должен создать файл напрямую по локальному пути и записать отчёт напрямую в Markdown-документ.

## Когда нужен перезапуск

После изменения файлов:

```text
C:\Users\REX\.codex\agents\simple_local.toml
C:\Users\REX\.config\opencode\opencode.jsonc
C:\Users\REX\.config\opencode\agents\qwen30b-local.md
C:\Users\REX\.config\opencode\agents\qwen480b-local.md
```

нужно перезапустить Codex и OpenCode, чтобы агенты заново подхватили конфигурацию.

Если после перезапуска агент всё ещё не пишет файлы, нужно повторить тест и смотреть его точный отчёт.

## Что должен делать основной агент

Если пользователь просит проверить или починить права субагентов, основной агент должен:

1. Прочитать этот документ.
2. Проверить `C:\Users\REX\.codex\agents\simple_local.toml`.
3. Проверить `C:\Users\REX\.config\opencode\opencode.jsonc`.
4. Проверить `C:\Users\REX\.config\opencode\agents\qwen30b-local.md`.
5. Проверить `C:\Users\REX\.config\opencode\agents\qwen480b-local.md`.
6. Если настройки неправильные, внести минимальную точечную правку.
7. Попросить пользователя перезапустить Codex и OpenCode.
8. После перезапуска дать тест локальному агенту `simple_local / qwen3-coder:30b / локальный`.
9. Если Codex-local не пишет, подготовить тест `qwen3-coder:30b / OpenCode Desktop / локальный через чат`.
10. Заполнить `Задача_внешнему_субагенту.md` для нужного OpenCode-агента.
11. Дать пользователю короткую задачу для вставки в OpenCode-чат.
12. После ответа пользователя прочитать `БЛОК 2` в `Задача_внешнему_субагенту.md`.
13. Вернуть итоговый отчёт с отдельными `SUBAGENT REPORT` по каждому агенту.

## Как тестировать локального агента

Основной агент запускает `simple_local / qwen3-coder:30b / локальный` сам.

Тест должен быть маленьким:

```text
Создай один тестовый Markdown-файл в явно указанной папке.
Не меняй другие файлы.
Работай локально на ПК, Git не используй вообще.
Верни отчёт с Agent, Status, Changed files, Result, Problems.
```

Если локальный агент не вернул отчёт или не смог писать, основной агент не скрывает это, а пишет отдельный `SUBAGENT REPORT`.

Если локальный агент отвечает, что `write_file` недоступен, это фиксируется как неуспешный тест записи. В этом случае основной агент не должен утверждать, что локальный субагент умеет создавать файлы через текущую Codex-subagent-среду.

## Как тестировать локального агента через OpenCode

Локальный агент `qwen3-coder:30b / OpenCode Desktop / локальный через чат` основной агент не запускает напрямую. Его тестируют так же, как внешнего OpenCode-агента: через рабочий документ и сообщение, которое пользователь вставляет в OpenCode-чат.

Этим агентом управляет только основной Codex. Если пользователь хочет задействовать `qwen30b-local`, он просит основной Codex поставить задачу. Самостоятельные задания локальному qwen30b вне рабочего документа не используются.

Локальный qwen30b можно держать готовым во время нашей работы, но только через controlled keep-warm: короткий warm-up без чтения и записи файлов. Агент должен ответить только `READY`, `Agent`, `Status`. Повторяющиеся псевдо-вызовы инструментов, долгие пустые рассуждения и работа без отчёта считаются сбоем, а не нормальным прогревом.

Основной агент:

1. Проверяет, что есть профиль `C:\Users\REX\.config\opencode\agents\qwen30b-local.md`.
2. Заполняет `M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_внешнему_субагенту.md`.
3. В `БЛОК 1` указывает `Agent: qwen3-coder:30b / OpenCode Desktop / локальный через чат`.
4. Разрешает создать только один тестовый файл и обновить только `БЛОК 2` рабочего документа.
5. Даёт пользователю короткий текст для вставки в OpenCode-чат.
6. После ответа пользователя читает `БЛОК 2`, проверяет созданный файл и пишет отдельный `SUBAGENT REPORT`.

Если локальный OpenCode-агент пишет файл успешно, это не отменяет проблему Codex-subagent. Это означает, что рабочий маршрут для записи локальной модели сейчас идёт через OpenCode Desktop.

## Как тестировать внешнего облачного агента

Внешний облачный агент `qwen3-coder:480b-cloud / OpenCode Desktop / внешний через чат` основной агент не запускает напрямую.

Основной агент:

1. Заполняет файл:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_внешнему_субагенту.md
```

2. В `БЛОК 1` пишет конкретную задачу, scope, разрешённые файлы, запреты и формат отчёта.
3. Даёт пользователю короткий текст для вставки в OpenCode-чат.
4. Пользователь вставляет этот текст внешнему облачному агенту.
5. Внешний агент выполняет задачу локально через OpenCode.
6. Внешний агент пишет полный отчёт в `БЛОК 2 — РАСШИРЕННЫЙ ОТЧЁТ ВНЕШНЕГО СУБАГЕНТА`.
7. Внешний агент возвращает пользователю короткий отчёт в чат OpenCode.
8. Пользователь копирует короткий отчёт основному агенту.
9. Основной агент читает `БЛОК 2`, проверяет результат и решает, что делать дальше.

## Формат короткого отчёта внешнего агента

```text
USER SUMMARY

Agent:
- qwen3-coder:480b-cloud / OpenCode Desktop / внешний через чат
- или qwen3-coder:30b / OpenCode Desktop / локальный через чат

Status:
- успешно / частично / неуспешно

Done:
- что сделано

Changed files:
- какие файлы изменены

For Codex:
- полный отчёт записан в БЛОК 2 файла Задача_внешнему_субагенту.md
```

## Главное правило

Субагенты помогают, но основной агент отвечает за scope, безопасность, проверку результата и финальное решение.

Если есть сомнение, субагенту дают read-only задачу или тест записи в один явно указанный файл.

Если задача основного агента успешно выполнена, основной агент сначала записывает итоговый отчёт в `БЛОК 2 — ПОСЛЕДНИЙ ОТЧЁТ АГЕНТА` рабочего документа `Задача_агенту.md`, а затем очищает `БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА`.

Если задача выполнена частично, неуспешно или требует решения пользователя, `БЛОК 1` не очищается, а причина указывается в `PROBLEMS`.

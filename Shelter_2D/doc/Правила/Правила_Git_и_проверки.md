# Правила Git и проверки

## Назначение

Документ описывает безопасную работу с Git, GitHub, diff, status, review и историей изменений в проекте `Shelter_2D`.

Репозиторий:

```text
https://github.com/maxrus82gm-svg/Shelter_2D.git
```

Политика Git:

```text
Git контролирует пользователь. Агент не делает commit, push, pull, merge, rebase, reset, clean, checkout, switch или изменение Git-конфига без прямой команды пользователя.
```

## Главное правило Git

Git контролирует пользователь.

Агент не делает commit, push, pull, merge, rebase, reset, clean, checkout, switch, создание веток, изменение remote, изменение Git-конфига или переписывание истории без прямой команды пользователя.

Read-only Git-команды разрешены только если текущая задача прямо разрешила Git-проверку.

Если текущая задача запрещает Git, не выполнять даже команды чтения.

## Review после отчёта агента

После каждого отчёта агента ChatGPT обязан выполнить review результата до итогового вывода пользователю.

Разделение обязанностей обязательно:

- агент-исполнитель не пишет блок `СТАТУС КОММИТА`;
- агент-исполнитель не указывает `Последний проверенный commit` как итоговый commit результата;
- агент-исполнитель не определяет актуальную верхушку репозитория для ChatGPT-review;
- агент-исполнитель может писать только `commit не создавался`, `push не выполнялся`, `Git-история не изменялась` и список выполненных read-only Git-проверок;
- если агент-исполнитель использует `git log -1`, это только baseline до работы или до пользовательского commit, а не итоговый commit результата;
- актуальный последний commit для review определяет только ChatGPT после отчёта агента, самостоятельно проверяя репозиторий или GitHub.

Порядок обязателен:

1. Сначала прочитать отчёт агента.
2. Затем проверить результат в репозитории или доступных файлах.
3. Самостоятельно определить актуальный последний commit, который проверяется.
4. Обязательно указать этот commit в начале review-ответа ChatGPT.
5. Сверить заявленные `DONE`, `CHANGED`, `DELETED`, `UNEXPECTED CHANGES`, `UNTRACKED`, `CHECKS`, `GIT`, `PROBLEMS`, `NEXT` с фактическим состоянием.
6. Проверить, что `БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА` очищен, если задача была завершена успешно.
7. Проверить, что отчёт записан в `БЛОК 2 — ПОСЛЕДНИЙ ОТЧЁТ АГЕНТА`.
8. Проверить, что не изменены запрещённые файлы и папки.
9. Только после проверки дать пользователю вывод.

ChatGPT не должен отвечать пользователю только на основании отчёта агента, если есть доступ к репозиторию или файлам для проверки.

Если проверка невозможна, ChatGPT обязан явно написать:

```text
Проверка репозитория не выполнена: <причина>.
```

После этого ChatGPT может дать только предварительный вывод, а не окончательное подтверждение.

Любой ChatGPT-review после отчёта агента начинается с:

```text
СТАТУС КОММИТА
Последний проверенный commit: <номер или short sha>
Commit message: <message>
Repository: <owner/repo>
Branch: <branch>
Проверка: выполнена / частично выполнена / не выполнена
```

Если последний commit определить невозможно, ChatGPT обязан написать:

```text
СТАТУС КОММИТА
Последний проверенный commit: НЕ ПРОВЕРЕН
Причина: <почему не удалось проверить>
Проверка: частично выполнена
```

Затем идут:

```text
GIT / REPOSITORY CHECK
REVIEW STATUS
FINDINGS
PROBLEMS
NEXT
```

Commit message указывается как есть. Если message `+`, `01`, `02`, `fix` или другой короткий текст, писать как есть.

Если message не подтверждён, писать `message не подтверждён`.

Если push не обнаружен, писать `push не обнаружен`.

GitHub не равен локальному Git: GitHub показывает только то, что уже попало в удалённый репозиторий.

## Служебные файлы как UNEXPECTED CHANGES

Если при GitHub / Git / diff / review-проверке обнаружены изменения в служебных файлах, которые не входили в scope задачи, они указываются в `UNEXPECTED CHANGES`.

Список служебных файлов берётся из паспорта проекта:

```text
.git; .vs; .obsidian; .gigacode; bin; obj
.vs/*; *.suo; *.user; *.vcxproj.user
```

Примеры служебных файлов:

- `.obsidian/*`;
- `.vscode/*`;
- workspace-файлы;
- session-файлы;
- cache-файлы;
- временные файлы IDE.

Примеры выше не означают, что такие папки есть в каждом проекте. Реальный список задаётся в паспорте проекта.

Запрещено писать служебные файлы как `UNCHANGED`, если проверенный diff показывает, что они изменялись.

Если Git / GitHub / diff не проверялись, агент не утверждает, что служебные файлы не менялись. В этом случае писать: `не проверялось`.

# Правила использования субагентов

Проект:

```text
Shelter_2D
```

Рабочая директория:

```text
m:\GitHub\Shelter_2D\Shelter_2D
```

Рабочий документ внешнего субагента:

```text
m:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_внешнему_субагенту.md
```

Основной агент:

```text
основной агент
```

Локальный агент:

```text
локальный агент
папка `Локальный агент`, если пользователь отдельно разрешит
```

Внешний агент:

```text
внешний агент через чат пользователя
пользователь передаёт малый блок во внешний чат агента
```

## КОНТРОЛЬ МАРШРУТА АГЕНТОВ

- Основной агент остаётся координатором, контролирует scope и принимает финальное решение.
- Внутренние Codex-агенты — это только агенты, запущенные средствами Codex.
- Локальный агент — отдельный локальный маршрут, например `simple_local`, если он доступен и пользователь не запретил его использование.
- Внешний агент — отдельный чат пользователя; основной агент подготавливает задачу, но не запускает этот чат сам.
- Универсальные chat-шаблоны передаются пользователю как отдельный маршрут, а не как запуск Codex-субагента.
- Маршруты нельзя смешивать; внешний запуск нельзя имитировать внутри основного агента.

## Главное правило

Основной агент принимает финальное решение, контролирует scope и проверяет результат.

Субагент не расширяет scope и выполняет только задачу, подготовленную основным агентом.

## Правило активного контролируемого делегирования

Основной агент не должен по умолчанию запрещать использование локального субагента или внешнего агента.

Перед нетривиальной задачей основной агент обязан кратко оценить, можно ли безопасно использовать:

- локального субагента `локальный агент` через `папка `Локальный агент`, если пользователь отдельно разрешит`;
- внешнего агента `внешний агент через чат пользователя` через `пользователь передаёт малый блок во внешний чат агента`;
- read-only разведку;
- независимую проверку;
- черновую подготовку структуры;
- сравнение документов;
- поиск несоответствий;
- проверку Markdown, UTF-8, placeholders, scope и отчётов.

Если делегирование может сэкономить ресурсы основного агента, уменьшить расход контекста, ускорить чтение или повысить качество проверки без потери контроля, основной агент должен рекомендовать или использовать такого помощника.

Основной агент не должен автоматически писать в задачах `Субагенты: Не использовать`, если нет явной причины.

Правильный подход по умолчанию:

```text
Субагенты:
Разрешены при безопасном и контролируемом scope. Основной агент сам оценивает, где использование локального субагента или внешнего агента может сэкономить ресурсы, ускорить проверку или дать независимый взгляд. Финальное решение остаётся за основным агентом.
```

Финальное решение, финальная проверка и ответственность всегда остаются за основным агентом.

## Правило экономного делегирования

Основной агент должен чаще рассматривать использование внешнего агента или локального субагента, если подзадачу можно безопасно ограничить и это уменьшает расход ресурсов основного агента.

Делегирование считается правильным только если оно:

- понятно основному агенту и пользователю;
- имеет короткий scope;
- не требует доверять субагенту финальное решение;
- не расширяет разрешённые файлы;
- не использует Git без отдельного разрешения;
- не замедляет задачу сильнее, чем экономит ресурсы;
- завершается понятным `SUBAGENT REPORT`.

Если подзадача слишком маленькая, рискованная, срочная, требует финального инженерного решения или может запутать отчёт, основной агент выполняет её сам и кратко указывает, что субагент не использовался.

## Когда субагента использовать желательно

Субагента или внешнего агента желательно использовать, если задача включает:

- чтение нескольких документов;
- сравнение рабочих правил и шаблона;
- поиск недостающих placeholders;
- поиск жёсткой привязки к конкретному проекту;
- проверку, не остались ли старые пути, названия, отчёты или артефакты;
- read-only анализ структуры;
- черновой список предложений;
- независимый review перед финальной правкой;
- проверку форматности Markdown;
- проверку, что файл не записан одной строкой;
- проверку, что нет текстовой последовательности "обратная косая черта + n" вместо переносов;
- проверку, что UTF-8 и кириллица сохранены.

## Когда субагента не использовать

Субагента не использовать или использовать только read-only, если задача:

- слишком маленькая и делегирование замедлит работу;
- требует срочного короткого ответа;
- требует финального инженерного решения;
- связана с рискованными правками;
- требует изменения Git, commit, push, pull, reset, clean;
- затрагивает служебные папки, реальные конфиги агентов или IDE из `.git; .vs; .obsidian; .gigacode; bin; obj; SFML-2.6.1; Локальный агент; Облачный агент` и `НЕ УКАЗАНО`;
- не имеет понятного scope;
- не имеет понятного критерия проверки;
- может запутать маршрут отчёта.

## Жёсткое разделение локального и внешнего маршрута

Локальный управляемый агент:

- это локальный агент, профиль или инструмент из паспорта проекта, которому основной агент может сам поставить задачу через `папка `Локальный агент`, если пользователь отдельно разрешит` и сам наблюдать результат;
- основной агент сам ставит ему задачу в доступный локальный чат или инструмент субагента;
- основной агент сам задаёт scope, запреты, формат отчёта и срок ожидания;
- основной агент сам проверяет результат локально;
- пользователь получает от основного агента краткий `SUBAGENT REPORT`.

Внешний агент через чат пользователя:

- это отдельная среда или чат, куда основной агент не пишет напрямую и куда задачу передаёт пользователь;
- основной агент не ставит ему задачу напрямую;
- основной агент сначала записывает задачу в `БЛОК 1` файла `m:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_внешнему_субагенту.md`;
- основной агент даёт пользователю короткое сообщение для чата внешнего агента: где читать задачу и куда писать расширенный отчёт;
- внешний агент пишет расширенный отчёт в `БЛОК 2` файла `m:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_внешнему_субагенту.md`;
- внешний агент дополнительно дублирует `USER SUMMARY` в свой чат для пользователя;
- пользователь пишет основному агенту, что внешний агент выполнил задачу, и приносит краткий отчёт;
- основной агент после этого читает `БЛОК 2`, проверяет результат и возвращает итоговый `SUBAGENT REPORT`.

Нельзя смешивать маршруты: если это локальная вкладка субагента, основной агент сам ставит задачу и сам проверяет результат; если это внешний чат пользователя, основной агент готовит рабочий документ и короткое сообщение для пользователя. Внешний агент не получает задачу без рабочего документа, кроме явно описанного резервного режима, а локальный управляемый агент не требует от пользователя ручного переноса отчёта между чатами, если основной агент имеет прямой локальный канал.

## Запрет имитации внешнего запуска

Если задача предназначена внешнему агенту через чат пользователя, основной агент не должен пытаться выполнить её сам как "внешний агент".

Основной агент обязан оставаться координатором:

- записать или проверить задачу в `БЛОК 1` файла `m:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_внешнему_субагенту.md`;
- дать пользователю короткое сообщение для внешнего чата;
- явно указать в сообщении, где внешнему агенту читать задачу;
- явно указать, куда внешнему агенту писать расширенный отчёт;
- попросить внешнего агента продублировать `USER SUMMARY` в своём чате для пользователя;
- ждать от пользователя сообщения, что внешний агент завершил задачу;
- только после этого читать `БЛОК 2`, проверять результат и писать `SUBAGENT REPORT`.

Основной агент не должен:

- называть себя внешним агентом;
- писать `AGENT EXTENDED REPORT` от имени внешнего агента;
- ставить внешнему агенту статус `успешно` или `неуспешно` вместо реального внешнего запуска;
- изменять `БЛОК 2` внешнего агента только потому, что внешний `БЛОК 1` пустой;
- ждать или создавать внешний чат внутри основного агента, если внешний агент работает только через чат пользователя.

Если `БЛОК 1` внешнего агента пустой, основной агент не имитирует внешний запуск. Он сообщает пользователю, что сначала нужно записать задачу в `БЛОК 1`, или сам записывает её туда, если это входит в текущий scope, а затем выдаёт короткое сообщение для передачи внешнему агенту.

## Preflight перед запуском внешнего агента

Перед тем как дать пользователю короткое сообщение для внешнего агента, основной агент обязан проверить:

1. задача действительно записана в `m:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_внешнему_субагенту.md`;
2. `БЛОК 1` внешнего агента не пустой;
3. задача не была ошибочно записана в `m:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md`;
4. короткое сообщение внешнему агенту указывает именно на `m:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_внешнему_субагенту.md`;
5. место для `AGENT EXTENDED REPORT` явно указано;
6. если `БЛОК 1` пустой, внешний запуск не начинать.

Если внешний task случайно оказался в `m:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md`, это ошибка маршрута. Основной агент обязан остановиться, не запускать внешнего агента и сначала исправить размещение задачи.

## Внешний или локальный агент через чат пользователя

Внешний или локальный агент может работать через чат пользователя в отдельной среде.

Порядок:

1. Основной агент пишет задачу и scope в рабочий документ.
2. Пользователь передаёт задачу в чат субагента.
3. Субагент выполняет только указанную задачу.
4. Субагент пишет `AGENT EXTENDED REPORT` в указанный `БЛОК 2`.
5. Субагент дублирует `USER SUMMARY` в свой чат для пользователя.
6. Основной агент проверяет результат и принимает решение.

## Резервный режим без рабочего документа

Обычный маршрут внешнего агента идёт через:

```text
m:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_внешнему_субагенту.md
```

Резервный режим разрешён только если текущая задача запрещает менять этот файл или он не входит в scope.

Условия резервного режима:

- пользователь явно разрешил использовать внешнего агента;
- основной агент сам пишет полный prompt;
- внешний агент получает задачу через чат пользователя;
- внешний агент не пишет в рабочий документ, который не входит в scope;
- отчёт возвращается в чат пользователя;
- основной агент пишет в `SUBAGENT REPORT`, что это был резервный режим без записи в `БЛОК 2`.

Резервный режим не обходит scope. Если текущая задача не разрешает внешнему агенту менять файлы, он работает только read-only.

## Запреты

- Субагент не использует Git без прямого разрешения.
- Субагент не очищает `БЛОК 1`, если это не разрешено задачей.
- Субагент не меняет файлы вне scope.
- Субагент не меняет служебные папки и реальные конфиги из паспорта проекта без разрешения: `.git; .vs; .obsidian; .gigacode; bin; obj; SFML-2.6.1; Локальный агент; Облачный агент`, `НЕ УКАЗАНО`.
- `Шаблон_агентской_системы` является площадкой сборки будущего единого документа-матрицы правил, но не рабочим источником правил текущего проекта; субагентам не дают документы из этой папки в обычных задачах.

## Финальный маркер

Субагент завершает работу маркером:

```text
РАБОТА ЗАВЕРШЕНА
Итог: успешно
Контроль возвращён основному агенту.
```

или:

```text
РАБОТА НЕ ЗАВЕРШЕНА
Итог: частично / неуспешно
Контроль возвращён основному агенту.
```

Без маркера отчёт считается неполным.

## Форматность Markdown

Субагент сохраняет Markdown как UTF-8 с настоящими переносами строк, не вставляет текстовую последовательность "обратная косая черта + n" и не экранирует Windows-пути двойными слэшами.

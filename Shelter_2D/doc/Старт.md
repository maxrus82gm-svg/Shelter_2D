## Что читать перед задачей

Перед началом любой задачи агент читает:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Старт.md
M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md
```

Перед нетривиальной задачей агент кратко оценивает, можно ли безопасно использовать внешнего агента или локального субагента для разведки, чтения, черновика, проверки или другой ограниченной подзадачи. Если это снижает расход ресурсов основного Codex и не ухудшает качество, контроль, скорость и понятность отчёта, агент предпочитает делегировать такую подзадачу по правилам субагентов.

Если задача требует тематических правил, агент читает только нужные документы из списка:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Список_правил.md
```

Если задача связана с Git, diff, status, review, commit, push, pull, ветками или историей изменений, агент дополнительно читает:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Правила_Git_и_Review.md
```

Если задача связана с созданием или изменением документации, агент дополнительно читает:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Правила_создания_документов.md
```

Если задача предполагает использование субагентов, делегирование, параллельную проверку или выбор между облачным и локальным агентом, агент дополнительно читает:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Правила_использования_субагентов.md
```

Если задача связана с правами субагентов, созданием файлов субагентами, настройкой Codex/OpenCode, permissions, sandbox или тестом записи, агент дополнительно читает:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Настройка_прав_субагентов.md
```

## Ветка субагентов в Obsidian

Ветка субагентов подключается к графу не напрямую из `Старт`, а через рабочую цепочку:

```text
Старт -> Задача_агенту -> Правила_использования_субагентов -> Настройка_прав_субагентов
```

Для чистого графа `Старт` держит только вход в рабочую задачу. Дальше `Задача_агенту` ведёт к истории задач и правилам использования субагентов, а `Правила_использования_субагентов` ведут в центральную настройку прав субагентов.

`Шаблон_агентской_системы` — это рабочая площадка для сборки будущего единого документа-матрицы правил. Сейчас матрица строится здесь на реальном примере `Shelter_2D`, но не является источником рабочих правил для реальной работы над `Shelter_2D`.

В обычных задачах проекта эту папку не читает и не использует ни основной агент, ни субагенты. Если папка будет удалена из проекта, рабочие правила `Shelter_2D` должны оставаться самодостаточными и не зависеть от её наличия.

Только если пользователь прямо поставил задачу создать, собрать, перенести, проверить, изменить или удалить будущий документ-матрицу правил, агент дополнительно читает документы шаблона строго по порядку:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Шаблон_агентской_системы\README_как_переносить.md
M:\GitHub\Shelter_2D\Shelter_2D\doc\Шаблон_агентской_системы\MANIFEST_структуры.md
M:\GitHub\Shelter_2D\Shelter_2D\doc\Шаблон_агентской_системы\00_Паспорт_проекта.md
M:\GitHub\Shelter_2D\Shelter_2D\doc\Шаблон_агентской_системы\02_Карта_переменных.md
M:\GitHub\Shelter_2D\Shelter_2D\doc\Шаблон_агентской_системы\01_Правило_подгонки_под_новый_проект.md
M:\GitHub\Shelter_2D\Shelter_2D\doc\Шаблон_агентской_системы\03_Чеклист_проверки_переноса.md
M:\GitHub\Shelter_2D\Shelter_2D\doc\Шаблон_агентской_системы\04_Dry_run_пример_переноса.md
```

Агент не должен начинать работу с частного шаблона, формы отчёта или отдельного правила внутри `Шаблон_агентской_системы`, пока не прочитаны README, manifest, паспорт проекта и карта переменных. Эти документы нельзя применять как рабочие правила текущего проекта, если задача не относится прямо к обслуживанию или переносу шаблона.

Агент читает только документы, нужные для текущей задачи. Запрещено читать весь Obsidian Vault или весь каталог `doc` подряд без необходимости.

## Правило завершения задачи

Если задача успешно выполнена, агент сначала записывает итоговый отчёт в `БЛОК 2 — ПОСЛЕДНИЙ ОТЧЁТ АГЕНТА` файла:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md
```

Затем агент очищает `БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА` до состояния:

```text
Активной задачи нет.
Предыдущая задача выполнена.
Ожидается новая задача от пользователя.
```

Если задача выполнена частично, неуспешно или требует решения пользователя, `БЛОК 1` не очищается, а причина указывается в `PROBLEMS`.

Если задача связана с review, проверкой отчёта, Git, GitHub, status, diff, commit, branch или результатом выполненной задачи, агент читает `Правила_Git_и_Review.md` и начинает ответ с блока `GIT / REPOSITORY CHECK`.

Если пользователь принёс отчёт Codex, OpenCode, главного агента или субагента о выполненной задаче, основной агент / ChatGPT сначала проверяет актуальный последний commit в GitHub, commit message, branch и short SHA. Ответ начинается с блока `СТАТУС КОММИТА`, затем идёт `GIT / REPOSITORY CHECK`, и только потом `REVIEW STATUS`.

## Правило Obsidian Vault

`Shelter_2D` — это Obsidian Vault проекта.

Заметки проекта создаются и связываются внутри Vault только по прямой задаче пользователя или по правилам документации.

Настройки Obsidian в `.obsidian` менять запрещено без прямой команды пользователя.

## Правило Git

Git полностью контролирует пользователь.

Агент не выполняет Git-операции, которые меняют состояние репозитория: `commit`, `push`, `pull`, `merge`, `rebase`, `reset`, `clean`, `checkout`, `switch`, создание веток, изменение remote, изменение Git-конфига или переписывание истории.

Для Git-задач агент следует документу:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Правила_Git_и_Review.md
```

## Правила C++ / кода

Без прямой задачи пользователя агент не меняет исходный код, настройки сборки, зависимости и структуру проекта.

Если меняются только Markdown-документы, сборка проекта не требуется.

## Правило пошагового обучения

Агент объясняет работу пошагово, если пользователь просит обучение или разбор.

Базовый порядок:

```text
маленькая задача
понятное изменение
проверка
отчёт
следующий шаг
```

Агент не перегружает пользователя лишними документами, командами и теорией без необходимости текущей задачи.

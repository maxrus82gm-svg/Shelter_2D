# Правила использования субагентов

## Назначение

Документ описывает, когда и как агент может использовать субагентов в проекте Shelter_2D.

Субагент помогает экономить контекст основного агента, разгружать локальный ПК и параллельно выполнять ограниченные задачи.

Главное правило: финальные решения, рискованные правки и итоговый отчёт остаются за основным агентом.

Основной агент жёстко контролирует scope субагента и не полагается только на то, что субагент сам найдёт нужные правила.

Все субагенты по умолчанию работают локально на ПК с файлами проекта через доступные инструменты Codex, OpenCode, VS Code или другой подключённой среды. Git для субагентов запрещён: не использовать Git для чтения, сравнения, поиска или изменения файлов, если пользователь отдельно не дал Git-задачу и не разрешил конкретные read-only Git-команды.

## Приоритет субагентов

Если в текущей среде доступен облачный субагент `qwen3-coder:480b-cloud`, использовать его как предпочтительный вариант для тяжёлых, но хорошо ограниченных задач.

Если `qwen3-coder:480b-cloud` недоступен как роль или инструмент субагента, не блокировать работу. В этом случае:

- для простых и низкорисковых задач можно использовать `simple_local` / `qwen3-coder:30b`;
- для проверки записи локальной модели можно использовать `qwen3-coder:30b / OpenCode Desktop / локальный через чат`;
- если локальная модель сильно нагружает память ПК, лучше продолжить основным агентом без субагента;
- если задача требует финального решения, основной агент выполняет её сам.

## Правило управления `qwen30b-local`

Локальный OpenCode-агент:

```text
qwen3-coder:30b / профиль qwen30b-local / OpenCode Desktop
```

используется только как управляемый субагент основного Codex.

Универсальный порядок:

1. Пользователь просит основного Codex поставить задачу локальному qwen30b.
2. Основной Codex сам формулирует задачу, scope, запреты и формат отчёта.
3. Основной Codex записывает задачу в рабочий документ или даёт пользователю точный текст для вставки в OpenCode-чат.
4. Пользователь может только передать подготовленную команду в OpenCode, если прямого канала у основного Codex нет.
5. Локальный qwen30b выполняет только подготовленную задачу.
6. Локальный qwen30b возвращает короткий отчёт пользователю и пишет расширенный отчёт в разрешённый документ.
7. Основной Codex читает отчёт, проверяет файлы, принимает или отклоняет результат и выводит `SUBAGENT REPORT` в основной чат.

Пользователь не ставит `qwen30b-local` самостоятельные задачи вне этой схемы. Если нужна задача для этого агента, пользователь просит основной Codex: "поставь задачу qwen30b-local". Основной Codex остаётся единственным постановщиком, контролёром scope и проверяющим результата.

## Правило прогрева локального qwen30b

Локальный qwen30b можно держать готовым во время работы основного Codex, потому что у локальной модели бывает долгий старт.

Правильный режим называется controlled keep-warm: короткая служебная задача без чтения файлов, без записи файлов, без Git и без вызова инструментов.

Если нужен прогрев модели или поддержание готовности, основной Codex даёт отдельную короткую warm-up задачу:

```text
Warm-up only. Ничего не читай и не меняй. Git не используй. Ответь только READY, Agent, Status.
```

Warm-up не должен иметь доступа к файлам, не должен создавать документы и не должен превращаться в бесконечную задачу.

Если локальный qwen30b:

- печатает tool-call текстом вместо реального вызова инструмента;
- повторяет один и тот же псевдо-вызов;
- пишет общие фразы без выполнения задачи;
- работает дольше установленного ожидания без отчёта;

основной Codex останавливает ожидание, фиксирует запуск как `неуспешно` или `не вернул отчёт` и пишет отдельный `SUBAGENT REPORT`. Такой режим не считается нормальным способом "не дать агенту уснуть".

Итог: держать модель тёплой можно; держать её в бесконечной имитации работы нельзя.

Проверенный статус для `qwen3-coder:30b / Codex subagent / local`: controlled keep-warm работает, но тест записи через Codex-subagent не создал файл. Для задач записи локальной qwen30b использовать маршрут OpenCode `qwen30b-local`, пока Codex-subagent не получит рабочий файловый инструмент.

## Внешний субагент через чат пользователя

Внешний субагент — это не конкретная модель, а управляемый маршрут работы через чат пользователя в другой среде. Это может быть `GigaChat / VS Code`, `qwen3-coder:480b-cloud / OpenCode Desktop`, другой облачный агент или локальная модель, если она имеет локальный доступ к файлам проекта.

Если внешний агент используется через VS Code, OpenCode Desktop или другую среду, основной агент не запускает его напрямую. Основной агент заполняет рабочий документ, а пользователь передаёт короткую команду в чат этой среды.

Рабочий документ для внешнего субагента:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_внешнему_субагенту.md
```

Порядок работы:

1. Основной агент пишет задачу во внешний документ в `БЛОК 1`.
2. Основной агент даёт пользователю короткое сообщение для чата внешнему субагенту.
3. Пользователь отправляет это сообщение в нужную среду внешнего агента: например VS Code, OpenCode Desktop или другой чат.
4. Внешний субагент выполняет задачу и даёт короткий отчёт пользователю.
5. Внешний субагент пишет расширенный отчёт в `БЛОК 2` или возвращает его пользователю текстом.
6. Основной агент читает расширенный отчёт, проверяет результат и возвращает в основной чат краткую выдержку как `SUBAGENT REPORT`.

Если внешний или локальный агент через чат пользователя не может записать отчёт в файл, пользователь приносит его расширенный отчёт в основной чат.

## Резервный режим внешнего агента без рабочего документа

Обычный маршрут внешнего агента всегда идёт через:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_внешнему_субагенту.md
```

Резервный режим разрешён только как исключение, если одновременно выполнены все условия:

- пользователь явно попросил или разрешил использовать внешнего агента;
- текущая задача запрещает менять `Задача_внешнему_субагенту.md` или этот файл не входит в разрешённый scope;
- использование внешнего агента не расширяет scope основной задачи;
- основной Codex сам формулирует полный prompt, ограничения, запреты и формат отчёта;
- внешний агент получает задачу только через чат пользователя;
- внешний агент не пишет в `БЛОК 2` рабочего документа, если этот документ не входит в scope;
- пользователь приносит `USER SUMMARY` и, если нужно, `AGENT EXTENDED REPORT` в основной чат;
- основной Codex проверяет результат и явно пишет в `SUBAGENT REPORT`, что агент работал в резервном режиме без записи в `БЛОК 2`.

Резервный режим не является обходом правил. Если задача требует записи отчёта внешним агентом в файл, но файл отчёта не входит в scope, внешний агент должен вернуть отчёт текстом в чат.

Если внешний агент в резервном режиме должен менять файлы, основной Codex обязан перечислить эти файлы в prompt. Если текущая задача не разрешает такие изменения, внешний агент используется только read-only или не используется.

## Рабочий контракт внешнего или локального агента

Любой внешний или локальный агент через чат пользователя работает только как исполнитель задачи, подготовленной основным Codex.

Перед началом агент обязан понять и явно отразить в отчёте:

- конкретное имя агента, модель, среду и маршрут работы;
- рабочий каталог;
- какие файлы разрешено читать;
- какие файлы или блоки разрешено менять;
- запрещён ли Git;
- куда писать расширенный отчёт.

Агент не должен расширять задачу самостоятельно. Если scope непонятен, файл недоступен, инструмент записи не работает или правила противоречат друг другу, агент останавливается и пишет `STATUS: неуспешно` или `STATUS: частично` с причиной в `PROBLEMS`.

Агент не очищает `БЛОК 1` рабочего документа самостоятельно, если это прямо не разрешено отдельной задачей. После успешного отчёта основной Codex проверяет результат и сам очищает активную задачу.

Агент должен сохранять Markdown как обычный UTF-8 текст:

- использовать настоящие переносы строк;
- не записывать весь файл одной строкой;
- не вставлять буквальные `\n` вместо переносов;
- не экранировать Windows-пути двойными слэшами;
- не превращать Markdown-документ в JSON-строку или строковый литерал.

Внешний или локальный агент обязан завершать `USER SUMMARY` явным маркером:

```text
РАБОТА ЗАВЕРШЕНА
Итог: успешно
Контроль возвращён основному Codex.
```

Если задача выполнена частично или не выполнена:

```text
РАБОТА НЕ ЗАВЕРШЕНА
Итог: частично / неуспешно
Контроль возвращён основному Codex.
```

Без такого маркера основной Codex считает отчёт неполным и проверяет результат особенно строго.

## Какие правила читает субагент

Субагент тоже должен знать правила проекта, но читать он должен только минимальный набор, нужный для своей подзадачи.

Основной агент обязан перед запуском субагента:

- явно указать, какие файлы правил прочитать;
- продублировать в prompt ключевые запреты и scope;
- разделить scope на чтение и правки;
- указать режим работы: read-only или edit;
- запретить выходить за разрешённые файлы и каталоги;
- запретить Git полностью для обычных подзадач;
- потребовать короткий отчёт о прочитанных правилах;
- потребовать два итоговых блока: `USER SUMMARY` и `AGENT EXTENDED REPORT`.

Для read-only подзадач обычно достаточно:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Старт.md
M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md
```

Для подзадач с документацией дополнительно читать:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Правила_создания_документов.md
M:\GitHub\Shelter_2D\Shelter_2D\doc\Правила_использования_субагентов.md
```

Git-задачи субагентам по умолчанию не передаются. Если пользователь отдельно разрешил Git-задачу для субагента, основной агент должен явно перечислить разрешённые read-only Git-команды и дополнительно дать прочитать:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Правила_Git_и_Review.md
```

Если субагенту разрешены правки, он обязан прочитать:

```text
M:\GitHub\Shelter_2D\Shelter_2D\doc\Старт.md
M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md
M:\GitHub\Shelter_2D\Shelter_2D\doc\Правила_использования_субагентов.md
```

И дополнительно тематические правила своей задачи.

Если субагент не имеет доступа к файлам или не может выполнить чтение, основной агент не должен блокироваться. В этом случае основной агент передаёт субагенту краткую выдержку правил прямо в prompt или выполняет задачу сам.

## Контроль основного агента

Основной агент отвечает за соблюдение правил даже тогда, когда работает субагент.

Основной агент обязан:

- проверить ответ субагента;
- не принимать вывод субагента без собственной проверки там, где есть риск;
- не отдавать субагенту финальное решение по рискованной задаче;
- не позволять субагенту расширять scope;
- остановить работу, если субагент предлагает действие вне правил;
- указать использование субагента в отчёте.

Субагент может помогать с анализом, поиском и черновиками, но основной агент принимает итоговое решение сам.

## Где полезен `qwen3-coder:480b-cloud`

Облачный субагент полезен для задач, которые требуют много чтения, сопоставления и чернового анализа:

- обзор больших участков кодовой базы;
- поиск архитектурных связей;
- сравнение нескольких вариантов решения;
- независимая проверка плана;
- черновики документации;
- разбор логов и отчётов;
- поиск причин бага по симптомам;
- подготовка краткой сводки по большому объёму материалов;
- ревью документации на противоречия;
- декомпозиция крупной задачи на безопасные шаги.

Облачный субагент особенно полезен, когда нужно экономить память локального ПК и не запускать тяжёлую локальную модель.

## Где лучше использовать `simple_local` / `qwen3-coder:30b`

Локальный `qwen3-coder:30b` доступен в текущей среде и может использоваться как локальный субагент.

Если он вызывается через роль `simple_local`, в отчёте указывать:

```text
Agent:
- simple_local / qwen3-coder:30b / локальный
```

Локальный `simple_local` / `qwen3-coder:30b` подходит для простых, низкорисковых задач:

- быстрый read-only поиск;
- краткая сводка по нескольким файлам;
- простые гипотезы по ошибке;
- черновой список мест, которые нужно прочитать;
- маленькие шаблоны текста или кода.

Если локальная модель заметно режет память ПК, её не использовать без необходимости.

## Что нельзя отдавать субагенту

Субагенту нельзя отдавать задачу без точного scope.

Субагент не должен:

- принимать финальные архитектурные решения вместо основного агента;
- использовать Git для обычной локальной работы с файлами;
- делать commit, push, pull, merge, rebase, reset, clean, checkout, switch;
- менять ветки;
- трогать код, ресурсы, конфиги проекта или рабочие файлы приложения без прямого разрешения задачи;
- менять `.obsidian` без прямой команды пользователя;
- читать всё хранилище подряд без необходимости;
- работать с секретами, токенами, приватными ключами и паролями.

Если задача может раскрыть чувствительные данные облачному агенту, основной агент обязан остановиться и уточнить разрешение у пользователя.

## Как формулировать задачу субагенту

Задача субагенту должна содержать:

- цель;
- какие правила нужно прочитать;
- разрешённые файлы или каталоги для чтения;
- разрешённые файлы для правок или прямое указание "ничего не менять";
- запреты;
- режим работы: read-only или можно редактировать;
- ожидаемый формат ответа;
- требование работать с локальными файлами на ПК без Git;
- требование не выходить за scope;
- чек-лист самопроверки перед ответом.

Пример:

```text
Прочитай только:
- M:\GitHub\Shelter_2D\Shelter_2D\doc\Старт.md
- M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md
Ничего не редактируй.
Работай локально с файлами на ПК, Git не используй.
Верни кратко: какие правила обязательны для текущей задачи и где есть противоречия.
```

## Правила для задач с правками

Если субагенту разрешены правки, основной агент обязан:

- дать субагенту отдельный, непересекающийся набор файлов;
- явно указать, что другие изменения в рабочем дереве нельзя откатывать;
- после результата проверить изменённые файлы;
- принять, доработать или отклонить результат сам;
- указать использование субагента в отчёте.

Субагент не должен править те же файлы параллельно с основным агентом, если это может создать конфликт.

## Отчёт о работе субагента

Если субагент использовался, основной агент обязан вернуть отчёт субагента в чат пользователя.

Если был задействован хотя бы один агент, основной агент всегда пишет отдельный краткий `SUBAGENT REPORT` по каждому агенту до основного отчёта. Отчёт должен быть самостоятельным: пользователь должен сразу видеть, к кому обращались, зачем, что агент сделал и успешно ли завершилась его часть работы.

В каждом отчёте обязательно указывается, к какому конкретно агенту обращался основной агент:

- локальный субагент внутри текущей среды, например `simple_local / qwen3-coder:30b`;
- внешний облачный агент через чат пользователя, например `qwen3-coder:480b-cloud в OpenCode Desktop`;
- внешний агент через VS Code, например `GigaChat / VS Code`;
- другой агент, если он был явно использован.

Нельзя писать только "субагент" без имени, роли, модели или среды. Пользователь всегда должен понимать, был ли это локальный агент, внешний облачный агент через чат или другой конкретный исполнитель.

Если основной агент только подготовил команду для внешнего облачного агента, но не запускал его напрямую, это тоже указывается в отчёте.

В кратком отчёте обязательно указывается итоговый статус работы агента:

- `успешно` — агент вернул полезный результат, который был использован;
- `частично` — агент вернул неполный результат или результат потребовал существенной проверки;
- `неуспешно` — агент не справился, ошибся или результат не использован;
- `не вернул отчёт` — агент был запущен, но не дал результата за время ожидания.

Отчёт в чат нужен даже тогда, когда субагент работал только read-only или ничего не смог сделать.

Есть два уровня отчётов:

- промежуточные лаконичные отчёты;
- итоговый отчёт по субагенту.

## Промежуточные лаконичные отчёты

Промежуточные отчёты нужны, чтобы пользователь понимал, что делает субагент и зачем он запущен.

Основной агент пишет промежуточный отчёт в чат:

- сразу после запуска субагента;
- если субагент вернул важный промежуточный результат;
- если субагент недоступен, завис, не смог прочитать файлы или не имеет нужного инструмента;
- если результат субагента меняет план основной работы.

Формат промежуточного отчёта:

```text
SUBAGENT UPDATE

Agent:
- конкретный агент, роль, модель и среда: например `simple_local / qwen3-coder:30b / локальный`, `GigaChat / VS Code / внешний через чат` или `qwen3-coder:480b-cloud / OpenCode Desktop / внешний через чат`

Task:
- коротко, зачем запущен

Status:
- запущен / работает / завершил / недоступен / ошибка

Current result:
- кратко, что уже понятно

Next:
- что основной агент делает дальше
```

Промежуточный отчёт должен быть коротким. Не нужно копировать длинный вывод субагента, если достаточно пересказать суть.

## Итоговый отчёт по субагенту

Итоговый отчёт фиксирует, что именно сделал субагент и как основной агент использовал результат.

В конце задачи порядок отчёта в чат всегда такой:

```text
SUBAGENT REPORT
- отчёт первого задействованного субагента

SUBAGENT REPORT
- отчёт второго задействованного субагента, если был

AGENT REPORT
- итоговый отчёт основного агента
```

Если субагентов было несколько, основной агент выводит отдельный короткий `SUBAGENT REPORT` по каждому из них.

Если вся задача основного агента успешно выполнена, основной агент сначала записывает итоговый отчёт в `БЛОК 2 — ПОСЛЕДНИЙ ОТЧЁТ АГЕНТА` рабочего документа `Задача_агенту.md`, а затем очищает `БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА`.

После очистки `БЛОК 1` должен содержать:

```text
# БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА

Активной задачи нет.
Предыдущая задача выполнена.
Ожидается новая задача от пользователя.
```

Если задача выполнена частично, неуспешно или требует решения пользователя, `БЛОК 1` не очищается, а причина указывается в `PROBLEMS`.

Если субагент запускался, но ничего полезного не сделал, его отчёт всё равно выводится. В нём указывается причина: недоступен, ошибка инструмента, не смог прочитать файлы, результат не использован.

Если субагент не вернул отчёт за время ожидания, был остановлен или закрыт основным агентом, это не скрывается. В `Result` указывается стандартная формулировка:

```text
Result:
- Субагент не вернул отчёт за время ожидания, был остановлен.
- Основной анализ выполнен мной локально через команды чтения.
```

Если после остановки основная работа была продолжена другим способом, основной агент кратко указывает, каким именно способом он получил результат.

Минимальный формат сообщения в чат:

```text
SUBAGENT REPORT

Agent:
- конкретный агент, роль, модель и среда

Purpose:
- зачем запускался

Status:
- успешно / частично / неуспешно / не вернул отчёт

Rules read:
- какие правила прочитал
- если не смог прочитать, почему

Scope:
- какие файлы или области были разрешены

Mode:
- read-only или edit

Result:
- что сделал или выяснил
- если отчёт не вернулся: указать, что конкретный агент не вернул отчёт за время ожидания, был остановлен, и как основной агент продолжил работу

Problems:
- ошибки, ограничения, недоступность инструмента

Used by main agent:
- как основной агент использовал результат
```

Если субагент вернул длинный ответ, основной агент кратко пересказывает его в чат и сохраняет важные детали: выводы, файлы, риски, ошибки и ограничения.

Если субагент не справился, это не скрывается. Основной агент пишет в чат, что субагент был недоступен, не смог прочитать файлы, не имел нужного инструмента или вернул непригодный результат.

Если субагент использовался, основной агент указывает в финальном отчёте:

```text
SUBAGENTS:
- какой субагент использовался
- зачем использовался
- режим: read-only или edit
- результат
- если субагент был недоступен, что использовано вместо него
```

Если субагент не использовался, но задача была простой, можно написать:

```text
SUBAGENTS:
- Не использовались; задача выполнена основным агентом.
```

## Главное правило

Субагент помогает, но не заменяет основного агента.

Основной агент отвечает за:

- понимание задачи пользователя;
- соблюдение scope;
- финальные правки;
- проверки;
- итоговый отчёт;
- безопасность проекта.

## Obsidian-граф

Граф должен быть логичным и читаемым.

Главная цепочка:

```text
Старт -> Задача_агенту -> Список_правил -> конкретные правила
```

`Старт.md` должен быть отдельной точкой входа.

`Старт.md` не должен ссылаться напрямую на все правила и все формы отчётов.

В `Старт.md` должна быть минимальная ссылка на:

```text
[[Задача_агенту]]
```

Можно оставить обычные пути в code block, но не делать из них wiki-ссылки, чтобы не засорять граф.

`Задача_агенту.md` должен ссылаться на:

```text
[[Список_правил]]
[[Список_форм_отчётов]]
```

`Список_правил.md` должен быть центром звезды для правил:

```text
[[README_как_переносить]]
[[MANIFEST_структуры]]
[[00_Паспорт_проекта]]
[[01_Правило_подгонки_под_новый_проект]]

[[04_Dry_run_пример_переноса]]

```

`Список_форм_отчётов.md` должен быть центром отдельной звезды для форм:

## Формат отчёта

В конце дать компактный отчёт:

```text
DONE:
- что сделано

CHANGED:
- намеренно изменённые файлы

UNEXPECTED CHANGES:
- побочные или автоматические изменения

UNTRACKED:
- новые неотслеживаемые файлы

CHECKS:
- что проверено

GIT:
- commit не создавался агентом
- push не выполнялся агентом
- pull не выполнялся агентом
- ветки не создавались и не переключались агентом
- Git-история не изменялась агентом

PROBLEMS:
- реальные проблемы или риски

NEXT:
- следующий шаг
```

# БЛОК 3 — REVIEW / STATUS

## Большой блок

Большой блок хранится в `{{MAIN_TASK_FILE}}`, содержит цель, контекст, scope, разрешённые файлы, запреты, проверки и формат отчёта.

## Малый блок

Малый блок отправляется агенту в чат и дополняет большой блок поздними уточнениями.

Если малый блок противоречит большому, агент останавливается и пишет конфликт в `PROBLEMS`.

Прямое сообщение пользователя имеет приоритет над задачей второго агента.

## Review после отчёта

После отчёта агента основной review начинается с:

```text
СТАТУС КОММИТА
GIT / REPOSITORY CHECK
```

Если GitHub недоступен, писать `message не подтверждён`.

Если изменения локальные и не запушены, писать `push не обнаружен`.

## Контроль размещения активной задачи

- задача основного агента пишется в `{{MAIN_TASK_FILE}}`;
- задача внешнего агента пишется в `{{EXTERNAL_AGENT_TASK_FILE}}`;
- если в `{{MAIN_TASK_FILE}}` найден заголовок `ТЕКУЩАЯ ЗАДАЧА ВНЕШНЕМУ СУБАГЕНТУ`, это ошибка структуры;
- если внешний агент получил короткое сообщение, но `БЛОК 1` внешнего документа пустой, запуск считается неподготовленным;
- основной агент не должен имитировать внешний запуск и не должен писать отчёт от имени внешнего агента.

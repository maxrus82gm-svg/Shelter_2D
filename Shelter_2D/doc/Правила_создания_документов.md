# Правила создания документов

## Назначение

Документ описывает правила создания и обновления Markdown-документов в проекте VR_Project.

Агент читает этот документ, если задача создаёт новый документ, меняет структуру документации, переносит правила, предлагает новое название документа или работает с Obsidian-заметками.

## Разделение `Doc` и `VR`

`Doc` — строгая система правил и задач агента внутри Obsidian Vault:

```text
M:\GITS_VERSE\VR_Project\VR_Projekt\VR\Doc
```

В `Doc` хранятся стартовые правила, задачи агенту, правила Git / Review и правила создания документов.

`VR` — Obsidian Vault проекта:

```text
M:\GITS_VERSE\VR_Project\VR_Projekt\VR
```

В `VR` хранятся проектные заметки, связанные материалы, рабочие записи и Obsidian-граф проекта.

Нельзя смешивать роли: агентские правила и активные задачи хранятся в `VR\Doc`, а обычные заметки Obsidian создаются в других частях `VR` только по прямой задаче пользователя.

## Правила создания документов в `Doc`

Новый документ в `Doc` создаётся только по прямой задаче пользователя или если текущая задача явно требует создать недостающий файл из списка разрешённых.

Перед созданием документа в `Doc` агент обязан:

- прочитать `M:\GITS_VERSE\VR_Project\VR_Projekt\VR\Doc\Старт.md`;
- прочитать `M:\GITS_VERSE\VR_Project\VR_Projekt\VR\Doc\Задача_агенту.md`;
- прочитать этот документ;
- проверить, что новый документ не дублирует уже существующий;
- убедиться, что путь входит в scope задачи.

Документы в `Doc` должны иметь:

- расширение `.md`;
- понятное русское название, если нет технической причины для другого имени;
- ясный заголовок;
- короткие разделы;
- актуальные пути проекта VR_Project;
- ссылку на GitVerse только в виде `git@gitverse.ru:MLPGM/VR_Project.git`, если ссылка нужна;
- UTF-8 без повреждения кириллицы.

Запрещено создавать дублирующие документы с тем же смыслом под новым именем. Если похожий документ уже есть, агент обновляет существующий документ только в рамках разрешённого scope.

## Правила создания заметок в Obsidian `VR`

Заметки в `VR` создаются только по прямой задаче пользователя или по явно описанной задаче на документацию проекта.

Перед созданием заметки агент должен:

- определить, зачем нужна заметка;
- проверить, что она не дублирует существующую;
- выбрать понятное русское название;
- сохранить файл как Markdown;
- не менять настройки Obsidian.

Если заметка связана с другой заметкой, использовать Obsidian-ссылки вида:

```text
двойные квадратные скобки вокруг названия заметки
```

Если нужна ссылка на заметку в подпапке, использовать формат, принятый в текущем Vault, и не добавлять лишние дубли.

## Запрет на `.obsidian`

Каталог `.obsidian` содержит настройки Obsidian.

Агенту запрещено менять `.obsidian` без прямой команды пользователя.

Если `.obsidian` изменился автоматически внешней программой, агент не исправляет это сам и указывает факт в отчёте, если изменение обнаружено в рамках проверки.

## Кодировка

Все Markdown-документы сохраняются в UTF-8.

Русский текст должен читаться нормально. Запрещены:

- вопросительные знаки вместо кириллицы;
- кракозябры;
- транслит вместо русского текста;
- нечитаемые символы из-за неверной кодировки.

Если после чтения текст выглядит повреждённым, агент останавливается и сообщает об этом в отчёте.

## Экономия контекста

Агент читает только документы, нужные для текущей задачи.

Запрещено читать весь `Doc` или весь Obsidian Vault `VR` подряд без необходимости.

## Отчёт после создания или обновления документа

После создания или обновления документа агент указывает:

```text
CHANGED FILES:
- путь

DIFF / SUMMARY:
- что изменено

ENCODING CHECK:
- UTF-8
- кириллица читается нормально
```

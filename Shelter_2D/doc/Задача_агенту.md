### 6.2. Обязательный формат начала review

В `Правила_Git_и_Review.md` нужно прописать обязательный формат начала review:

```text
СТАТУС КОММИТА: "<commit message>"
Repository: owner/repo
Branch: main
Commit: <short sha>
Проверка: изменения найдены / push не обнаружен / требуется уточнение
```

Важно:

- этот блок должен идти самым первым;
    
- нельзя сначала писать “отлично”, “принято”, “выглядит нормально”;
    
- нельзя принимать отчёт только по словам агента;
    
- сначала проверяется последний commit;
    
- потом пишется `GIT / REPOSITORY CHECK`;
    
- только потом пишется `REVIEW STATUS`.
    

---

### 6.3. Правило commit message

В `Правила_Git_и_Review.md` нужно прописать:

```text
Если commit message доступен, агент указывает именно его.
```

Если commit message короткий или неинформативный, например:

```text
+
01
02
fix
test
```

агент всё равно указывает его как есть.

Запрещено:

- улучшать commit message;
    
- переписывать commit message;
    
- додумывать commit message;
    
- заменять короткий message на красивое описание;
    
- скрывать неинформативный message.
    

Правильный пример:

```text
СТАТУС КОММИТА: "+"
Repository: maxrus82gm-svg/Shelter_2D
Branch: main
Commit: a1b2c3d
Проверка: изменения найдены
```

---

### 6.4. Если push не обнаружен

Добавить правило:

Если пользователь принёс отчёт о выполненной задаче, но в GitHub последний commit не содержит заявленных изменений, агент пишет:

```text
СТАТУС КОММИТА: push не обнаружен
Repository: owner/repo
Branch: main
Commit: <short sha последнего доступного commit>
Проверка: изменения не подтверждены через GitHub
```

После этого агент должен пояснить:

```text
Изменения могут быть только локально на ПК пользователя.
Через GitHub они пока не подтверждены.
Для полной проверки нужен push, commit SHA, git status, git diff или содержимое изменённых файлов.
```

---

### 6.5. Если commit message не подтверждён

Добавить правило:

Если commit message не удалось подтвердить, агент пишет:

```text
СТАТУС КОММИТА: message не подтверждён
Repository: owner/repo
Branch/ref: <branch/ref или не подтверждён>
Commit: <short sha или не подтверждён>
Проверка: требуется уточнение
```

Если не подтверждён даже commit SHA:

```text
СТАТУС КОММИТА: message не подтверждён
Repository: owner/repo
Branch/ref: не подтверждён
Commit: не подтверждён
Проверка: требуется уточнение
```

---

### 6.6. Связь с `GIT / REPOSITORY CHECK`

Нужно явно прописать:

```text
Блок `СТАТУС КОММИТА` не заменяет `GIT / REPOSITORY CHECK`.
```

Правильный порядок review:

```text
СТАТУС КОММИТА
GIT / REPOSITORY CHECK
REVIEW STATUS
FINDINGS
PROBLEMS
NEXT
```

Пример:

```text
СТАТУС КОММИТА: "02"
Repository: maxrus82gm-svg/Shelter_2D
Branch: main
Commit: a1b2c3d
Проверка: изменения найдены

GIT / REPOSITORY CHECK

Repository:
- проверен GitHub: maxrus82gm-svg/Shelter_2D

Branch:
- проверена ветка: main

Status:
- локальный status не проверялся
- GitHub-содержимое проверено по последнему commit

Changed files:
- проверены по GitHub / commit / отчёту агента

Diff summary:
- кратко описать, что изменилось

Commit:
- a1b2c3d
- message: "02"

Push:
- не выполнялся основным агентом
- наличие изменений в GitHub подтверждает, что push уже был выполнен пользователем или другим агентом

Pull:
- не выполнялся основным агентом

GitHub check:
- проверялся
```

---

### 6.7. Разделить GitHub и локальный Git

Нужно прописать важное ограничение:

```text
GitHub показывает только то, что уже попало в репозиторий.
```

Основной агент не должен утверждать, что проверил локальные незакоммиченные изменения, если нет:

```text
git status
git diff
git log -1
содержимого изменённых файлов
```

Если изменения есть только локально и не запушены, GitHub-проверка должна завершиться статусом:

```text
Проверка: push не обнаружен
```

Файлы разделяются так:

- `CHANGED` — намеренно изменённые файлы в рамках задачи;
- `UNEXPECTED CHANGES` — побочные, автоматические или не входившие в задачу изменения;
- `UNTRACKED` — новые неотслеживаемые файлы, если Git-проверка была разрешена и выполнялась.

Побочные изменения нельзя скрывать. Если, например, `.obsidian\workspace.json` изменился автоматически из-за Obsidian, но задача не разрешала менять его намеренно, файл указывается в `UNEXPECTED CHANGES`, если изменение обнаружено в рамках разрешённых проверок.

Формат отчёта:

После отчёта агента review начинается с проверки последнего commit message.

Обязательный порядок:

```text
СТАТУС КОММИТА
GIT / REPOSITORY CHECK
REVIEW STATUS
FINDINGS
PROBLEMS
NEXT
```

Любой review / проверка отчёта после выполненной задачи начинается с блока GIT / REPOSITORY CHECK.

Если Git не использовался, всё равно нужно явно написать:

```text
GIT / REPOSITORY CHECK

Branch:
- не проверялась

Status:
- не проверялся

Changed files:
- по отчёту агента

Diff summary:
- не проверялся

Commit:
- не создавался агентом

Push:
- не выполнялся агентом

Pull:
- не выполнялся агентом

GitHub check:
- не проверялся
```

```text
DONE:
- что сделано

CHANGED:
- намеренно изменённые файлы

UNEXPECTED CHANGES:
- побочные или автоматические изменения

UNTRACKED:
- новые неотслеживаемые файлы или "не проверялось Git-командами"

CHECKS:
- что проверено

GIT:
- использовался или не использовался
- commit не создавался агентом
- push не выполнялся агентом
- pull не выполнялся агентом
- ветки не создавались и не переключались агентом
- Git-история не изменялась агентом

PROBLEMS:
- реальные проблемы или риски

NEXT:
- следующий шаг
```

## Правило очистки активной задачи после отчёта

Если задача успешно выполнена, агент сначала записывает итоговый отчёт в `БЛОК 2 — ПОСЛЕДНИЙ ОТЧЁТ АГЕНТА`, а затем очищает `БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА`.

После очистки `БЛОК 1` должен содержать только:

```text
# БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА

Активной задачи нет.
Предыдущая задача выполнена.
Ожидается новая задача от пользователя.
```

Если задача выполнена частично, неуспешно, есть критичная ошибка или требуется решение пользователя, `БЛОК 1` не очищать. В этом случае агент обязан явно указать причину в `PROBLEMS`.

Очистка `БЛОКА 1` выполняется только после того, как отчёт записан в `БЛОК 2 — ПОСЛЕДНИЙ ОТЧЁТ АГЕНТА`.

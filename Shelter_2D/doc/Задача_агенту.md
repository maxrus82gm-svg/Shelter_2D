# ПРАВИЛО БОЛЬШОГО И МАЛОГО БЛОКА

Задачи могут приходить от пользователя напрямую или через второго агента, который готовит грамотную постановку задачи.

Задача второго агента не отменяет правила проекта и прямые сообщения пользователя.

## Большой блок

Большой блок:

- хранится в `M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md`;
- остаётся основной задачей в документе;
- содержит цель, контекст, scope, разрешённые файлы, запреты и проверки;
- не переписывается заново только из-за поздних уточнений.

## Малый блок

Малый блок:

- отправляется агенту в чат;
- содержит команду прочитать актуальную задачу из `M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md`;
- может содержать список дополнительных документов для чтения;
- может содержать уточнения, появившиеся после подготовки большого блока;
- может содержать критичные напоминания;
- дополняет большой блок, но не заменяет его.

Малый блок может быть написан пользователем или подготовлен вторым агентом.

Если уточнение появилось после подготовки большого блока и до отчёта агента, оно добавляется в малый блок без переписывания большого блока.

Если малый блок противоречит большому блоку, агент обязан остановиться и указать конфликт в отчёте.

Если задача второго агента противоречит прямому сообщению пользователя, приоритет имеет прямое сообщение пользователя.

# КОМПАКТНЫЙ ОТЧЁТ АГЕНТА

Последний отчёт агента хранится в `M:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md` в блоке `БЛОК 2 — ПОСЛЕДНИЙ ОТЧЁТ АГЕНТА`, ниже активной задачи.

Перед записью отчёта агент сверяет фактическое состояние файлов через разрешённые проверки. Если текущая задача запрещает Git, проверки выполняются без Git.

Если задача разрешает Git-проверки чтения и они показывают файл как изменённый, добавленный, удалённый или untracked, этот файл обязательно указывается в отчёте.

Файлы разделяются так:

- `CHANGED` — намеренно изменённые файлы в рамках задачи;
- `UNEXPECTED CHANGES` — побочные, автоматические или не входившие в задачу изменения;
- `UNTRACKED` — новые неотслеживаемые файлы, если Git-проверка была разрешена и выполнялась.

Побочные изменения нельзя скрывать. Если, например, `.obsidian\workspace.json` изменился автоматически из-за Obsidian, но задача не разрешала менять его намеренно, файл указывается в `UNEXPECTED CHANGES`, если изменение обнаружено в рамках разрешённых проверок.

Формат отчёта:

Любой review / проверка отчёта после выполненной задачи начинается с блока GIT / REPOSITORY CHECK.

Если Git не использовался, всё равно нужно явно написать:

```text
GIT / REPOSITORY CHECK

Branch:
- не проверялась

Status:
- не проверялся

Changed files:
- по отчёту агента

Diff summary:
- не проверялся

Commit:
- не создавался агентом

Push:
- не выполнялся агентом

Pull:
- не выполнялся агентом

GitHub check:
- не проверялся
```

```text
DONE:
- что сделано

CHANGED:
- намеренно изменённые файлы

UNEXPECTED CHANGES:
- побочные или автоматические изменения

UNTRACKED:
- новые неотслеживаемые файлы или \"не проверялось Git-командами\"

CHECKS:
- что проверено

GIT:
- использовался или не использовался
- commit не создавался агентом
- push не выполнялся агентом
- pull не выполнялся агентом
- ветки не создавались и не переключались агентом
- Git-история не изменялась агентом

PROBLEMS:
- реальные проблемы или риски

NEXT:
- следующий шаг
```

## Правило очистки активной задачи после отчёта

Если задача успешно выполнена, агент сначала записывает итоговый отчёт в `БЛОК 2 — ПОСЛЕДНИЙ ОТЧЁТ АГЕНТА`, а затем очищает `БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА`.

После очистки `БЛОК 1` должен содержать только:

```text
# БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА

Активной задачи нет.
Предыдущая задача выполнена.
Ожидается новая задача от пользователя.
```

Если задача выполнена частично, неуспешно, есть критичная ошибка или требуется решение пользователя, `БЛОК 1` не очищать. В этом случае агент обязан явно указать причину в `PROBLEMS`.

Очистка `БЛОКА 1` выполняется только после того, как отчёт записан в `БЛОК 2 — ПОСЛЕДНИЙ ОТЧЁТ АГЕНТА`.

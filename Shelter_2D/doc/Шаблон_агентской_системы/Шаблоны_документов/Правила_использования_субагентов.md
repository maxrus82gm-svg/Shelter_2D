# Правила использования субагентов

Проект:

```text
{{PROJECT_NAME}}
```

Рабочая директория:

```text
{{PROJECT_ROOT}}
```

Рабочий документ внешнего субагента:

```text
{{DOC_FOLDER}}\Задача_внешнему_субагенту.md
```

## Главное правило

Основной агент принимает финальное решение, контролирует scope и проверяет результат.

Субагент не расширяет scope и выполняет только задачу, подготовленную основным агентом.

## Правило экономного делегирования

Основной агент должен чаще рассматривать использование внешнего агента или локального субагента, если подзадачу можно безопасно ограничить и это уменьшает расход ресурсов основного агента.

Предпочтительные подзадачи для делегирования:

- read-only разведка по файлам;
- поиск по документам и коду;
- краткое сравнение нескольких документов;
- черновик структуры или текста;
- независимая проверка уже сделанной правки;
- подготовка вариантов решения без финального решения;
- проверка форматности Markdown, UTF-8, явных маркеров отчёта и похожих механических условий.

Делегирование считается правильным только если оно:

- понятно основному агенту и пользователю;
- имеет короткий scope;
- не требует доверять субагенту финальное решение;
- не расширяет разрешённые файлы;
- не использует Git без отдельного разрешения;
- не замедляет задачу сильнее, чем экономит ресурсы;
- завершается понятным `SUBAGENT REPORT`.

Если подзадача слишком маленькая, рискованная, срочная, требует финального инженерного решения или может запутать отчёт, основной агент выполняет её сам и кратко указывает, что субагент не использовался.

## Жёсткое разделение локального и внешнего маршрута

Локальный управляемый агент:

- это локальный чат или вкладка субагента внутри Codex/OpenCode, например `qwen3-coder-30b`, `Scout`, `simple_local` или аналогичный локальный профиль, если основной агент может сам поставить задачу в этот чат и наблюдать результат;
- основной агент сам ставит ему задачу в доступный локальный чат или инструмент субагента;
- основной агент сам задаёт scope, запреты, формат отчёта и срок ожидания;
- основной агент сам проверяет результат локально;
- пользователь получает от основного агента краткий `SUBAGENT REPORT`.

Внешний агент через чат пользователя:

- это отдельная среда или чат, куда основной агент не пишет напрямую и куда задачу передаёт пользователь;
- основной агент не ставит ему задачу напрямую;
- основной агент сначала записывает задачу в `БЛОК 1` файла `Задача_внешнему_субагенту.md`;
- основной агент даёт пользователю короткое сообщение для чата внешнего агента: где читать задачу и куда писать расширенный отчёт;
- внешний агент пишет расширенный отчёт в `БЛОК 2` файла `Задача_внешнему_субагенту.md`;
- внешний агент дополнительно дублирует `USER SUMMARY` в свой чат для пользователя;
- пользователь пишет основному агенту, что внешний агент выполнил задачу, и приносит краткий отчёт;
- основной агент после этого читает `БЛОК 2`, проверяет результат и возвращает итоговый `SUBAGENT REPORT`.

Нельзя смешивать маршруты: если это локальная вкладка субагента, основной агент сам ставит задачу и сам проверяет результат; если это внешний чат пользователя, основной агент готовит рабочий документ и короткое сообщение для пользователя. Внешний агент не получает задачу без рабочего документа, кроме явно описанного резервного режима, а локальный управляемый агент не требует от пользователя ручного переноса отчёта между чатами, если основной агент имеет прямой локальный канал.

## Запрет исполнять внешний маршрут внутри основного агента

Если задача предназначена внешнему агенту через чат пользователя, основной агент не должен пытаться выполнить её сам как "внешний агент".

Основной агент обязан оставаться координатором:

- записать или проверить задачу в `БЛОК 1` файла `Задача_внешнему_субагенту.md`;
- дать пользователю короткое сообщение для внешнего чата;
- явно указать в сообщении, где внешнему агенту читать задачу;
- явно указать, куда внешнему агенту писать расширенный отчёт;
- попросить внешнего агента продублировать `USER SUMMARY` в своём чате для пользователя;
- ждать от пользователя сообщения, что внешний агент завершил задачу;
- только после этого читать `БЛОК 2`, проверять результат и писать `SUBAGENT REPORT`.

Основной агент не должен:

- называть себя внешним агентом;
- писать `AGENT EXTENDED REPORT` от имени внешнего агента;
- ставить внешнему агенту статус `успешно` или `неуспешно` вместо реального внешнего запуска;
- изменять `БЛОК 2` внешнего агента только потому, что внешний `БЛОК 1` пустой;
- ждать или создавать внешний чат внутри основного агента, если внешний агент работает только через чат пользователя.

Если `БЛОК 1` внешнего агента пустой, основной агент не имитирует внешний запуск. Он сообщает пользователю, что сначала нужно записать задачу в `БЛОК 1`, или сам записывает её туда, если это входит в текущий scope, а затем выдаёт короткое сообщение для передачи внешнему агенту.

## Внешний или локальный агент через чат пользователя

Внешний или локальный агент может работать через чат пользователя в отдельной среде.

Порядок:

1. Основной агент пишет задачу и scope в рабочий документ.
2. Пользователь передаёт задачу в чат субагента.
3. Субагент выполняет только указанную задачу.
4. Субагент пишет `AGENT EXTENDED REPORT` в указанный `БЛОК 2`.
5. Субагент дублирует `USER SUMMARY` в свой чат для пользователя.
6. Основной агент проверяет результат и принимает решение.

## Резервный режим без рабочего документа

Обычный маршрут внешнего агента идёт через:

```text
{{DOC_FOLDER}}\Задача_внешнему_субагенту.md
```

Резервный режим разрешён только если текущая задача запрещает менять этот файл или он не входит в scope.

Условия резервного режима:

- пользователь явно разрешил использовать внешнего агента;
- основной агент сам пишет полный prompt;
- внешний агент получает задачу через чат пользователя;
- внешний агент не пишет в рабочий документ, который не входит в scope;
- отчёт возвращается в чат пользователя;
- основной агент пишет в `SUBAGENT REPORT`, что это был резервный режим без записи в `БЛОК 2`.

Резервный режим не обходит scope. Если текущая задача не разрешает внешнему агенту менять файлы, он работает только read-only.

## Запреты

- Субагент не использует Git без прямого разрешения.
- Субагент не очищает `БЛОК 1`, если это не разрешено задачей.
- Субагент не меняет файлы вне scope.
- Субагент не меняет `.git`, `.obsidian` и реальные конфиги без разрешения.
- `Шаблон_агентской_системы` является площадкой сборки будущего единого документа-матрицы правил, но не рабочим источником правил текущего проекта; субагентам не дают документы из этой папки в обычных задачах.

## Финальный маркер

Субагент завершает работу маркером:

```text
РАБОТА ЗАВЕРШЕНА
Итог: успешно
Контроль возвращён основному Codex.
```

или:

```text
РАБОТА НЕ ЗАВЕРШЕНА
Итог: частично / неуспешно
Контроль возвращён основному Codex.
```

Без маркера отчёт считается неполным.

## Форматность Markdown

Субагент сохраняет Markdown как UTF-8 с настоящими переносами строк, не вставляет буквальные `\n` и не экранирует Windows-пути двойными слэшами.

## Что читать перед задачей

Перед началом любой задачи агент читает:

```text
m:\GitHub\Shelter_2D\Shelter_2D\doc\Старт.md
m:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md
```

Для тематических правил агент открывает список правил по пути:

```text
m:\GitHub\Shelter_2D\Shelter_2D\doc\Список_правил.md
```

`Старт.md` остаётся короткой точкой входа. Он не заменяет список правил и не содержит прямых ссылок на все правила или формы отчётов.

## Правило завершения задачи

Если задача успешно выполнена, агент сначала записывает итоговый отчёт в `БЛОК 2 — ПОСЛЕДНИЙ ОТЧЁТ АГЕНТА` файла:

```text
m:\GitHub\Shelter_2D\Shelter_2D\doc\Задача_агенту.md
```

Затем агент очищает `БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА` до состояния:

```text
Активной задачи нет.
Предыдущая задача выполнена.
Ожидается новая задача от пользователя.
```

Если задача выполнена частично, неуспешно или требует решения пользователя, `БЛОК 1` не очищается, а причина указывается в `PROBLEMS`.

## Правило review после отчёта агента

После каждого отчёта агента ChatGPT обязан сразу проверить результат в репозитории или доступных файлах и только потом давать пользователю итоговый вывод.

Это правило относится к ChatGPT-review после отчёта агента.

Агент-исполнитель в своём итоговом отчёте:

- не пишет блок `СТАТУС КОММИТА`;
- не указывает `Последний проверенный commit` как итоговый commit результата;
- не определяет актуальную верхушку репозитория для ChatGPT-review;
- может писать только `commit не создавался`, `push не выполнялся`, `Git-история не изменялась` и список выполненных read-only Git-проверок.

Если агент использует `git log -1`, этот результат считается только baseline до работы или до пользовательского commit, а не итоговым commit результата.

Актуальный последний commit для review определяет только ChatGPT после отчёта агента, самостоятельно проверяя репозиторий или GitHub.

Порядок обязателен:

1. Прочитать отчёт агента.
2. Проверить результат в репозитории или доступных файлах.
3. Самостоятельно определить актуальный последний commit, который проверяется.
4. Указать этот commit в начале review-ответа ChatGPT.
5. Сверить `DONE`, `CHANGED`, `DELETED`, `UNEXPECTED CHANGES`, `UNTRACKED`, `CHECKS`, `GIT`, `PROBLEMS`, `NEXT` с фактическим состоянием.
6. Проверить, что `БЛОК 1 — ТЕКУЩАЯ ЗАДАЧА` очищен после успешной задачи.
7. Проверить, что отчёт записан в `БЛОК 2 — ПОСЛЕДНИЙ ОТЧЁТ АГЕНТА`.
8. Проверить, что запрещённые файлы и папки не менялись.
9. Только после проверки дать пользователю вывод.

Формат начала review-ответа:

```text
СТАТУС КОММИТА
Последний проверенный commit: <номер или short sha>
Commit message: <message>
Repository: <owner/repo>
Branch: <branch>
Проверка: выполнена / частично выполнена / не выполнена
```

Если последний commit определить невозможно, ChatGPT начинает review так:

```text
СТАТУС КОММИТА
Последний проверенный commit: НЕ ПРОВЕРЕН
Причина: <почему не удалось проверить>
Проверка: частично выполнена
```

Если проверка невозможна, ChatGPT пишет причину и даёт только предварительный вывод.

## Правило Git

Git контролирует пользователь.

Политика Git:

```text
Git контролирует пользователь. Агент не делает commit, push, pull, merge, rebase, reset, clean, checkout, switch или изменение Git-конфига без прямой команды пользователя.
```

Агент не делает commit, push, pull, merge, rebase, reset, clean, checkout, switch или изменение Git-конфига без прямой команды пользователя.

## Правило Obsidian Vault

Настройки `.obsidian` не менять без прямого разрешения пользователя. Граф связывается только Markdown-ссылками.
